2. Open-Source Trading Frameworks

2.1 QuantConnect / Lean Engine

  • What: Open-source algorithmic trading engine (C# with Python bindings)
  • Maturity: Very high — production-grade, powers QuantConnect cloud platform
  • Community: 9k+ GitHub stars, large community, active development
  • Key Features: Multi-asset (equities, options, futures, crypto), cloud backtesting, live trading, data library
  • Limitations: C# core (Python is secondary), cloud-dependent for full features, steep learning curve
  • Integration: Could use Lean locally for backtesting; complex to embed as a skill

2.2 Backtrader

  • What: Python backtesting library for trading strategies
  • Maturity: Mature but maintenance has slowed
  • Community: 13k+ GitHub stars
  • Key Features: Event-driven, supports multiple data feeds, live trading via IB, extensive indicator library
  • Limitations: Development largely stalled since ~2021, single-threaded, slow for large datasets
  • Integration: Good for prototyping strategies; can be used as backtesting engine within our skill

2.3 Zipline / Zipline-Reloaded

  • What: Pythonic algorithmic trading library (originally by Quantopian)
  • Maturity: Original abandoned; zipline-reloaded maintained by community
  • Community: 17k+ stars (original), ~1k (reloaded)
  • Key Features: Pipeline API for factor models, minute-level backtesting, Quantopian ecosystem
  • Limitations: US equities only, no live trading built-in, data ingestion can be painful
  • Integration: Pipeline API is excellent for factor-based strategies

2.4 VectorBT

  • What: Vectorized backtesting library using NumPy/Pandas
  • Maturity: Active development, modern design
  • Community: 4k+ GitHub stars
  • Key Features: Extremely fast backtesting (vectorized), portfolio optimization, rich visualization, parameter optimization
  • Limitations: Less intuitive for event-driven strategies, Pro version paywalled
  • Integration: Best candidate for fast backtesting — vectorized approach ideal for an AI agent iterating on strategies

2.5 Freqtrade

  • What: Python crypto trading bot framework
  • Maturity: Very mature, active development
  • Community: 30k+ GitHub stars, very active Discord
  • Key Features: Strategy development, backtesting, paper trading, live trading, web UI, hyperopt
  • Limitations: Crypto-focused — no native stock support
  • Integration: Architecture patterns worth studying but not directly usable for US stocks

2.6 Jesse

  • What: Python crypto trading framework
  • Maturity: Good, active development
  • Key Features: Clean strategy API, backtesting, live trading
  • Limitations: Crypto only
  • Integration: Limited for our use case

2.7 Hummingbot

  • What: Open-source market-making / algo trading bot
  • Community: 8k+ GitHub stars
  • Limitations: Primarily crypto/DEX focused
  • Integration: Market-making strategies could be adapted conceptually

3. AI/LLM Trading Agents (GitHub Projects)

3.1 TradingAgents (TauricResearch)

  • URL: github.com/TauricResearch/TradingAgents
  • What: Multi-agent LLM trading framework mirroring real trading firms
  • Architecture: Specialized agents — fundamental analyst, sentiment analyst, technical analyst, trader, risk manager, fund manager
  • Key Features: Debate-driven decision making, multi-agent collaboration, news/sentiment integration
  • Maturity: Research project (arxiv paper 2412.20138), not production-ready
  • Integration: Excellent architectural reference for our multi-agent design

3.2 FinRL (AI4Finance-Foundation)

  • URL: github.com/AI4Finance-Foundation/FinRL
  • What: Deep Reinforcement Learning framework for financial trading
  • Community: 10k+ GitHub stars
  • Key Features: DRL algorithms for portfolio allocation, market environments (gym-compatible), multi-asset
  • Maturity: Academic project, well-maintained
  • Limitations: RL approach (not LLM), requires significant training data/compute
  • Integration: RL-based strategy generation could complement LLM reasoning

3.3 FinGPT (AI4Finance-Foundation)

  • URL: github.com/AI4Finance-Foundation/FinGPT
  • What: Open-source financial LLMs
  • Community: 14k+ GitHub stars
  • Key Features: Fine-tuned LLMs for financial tasks (sentiment, NER, Q&A), data pipelines for financial data
  • Maturity: Active research project
  • Limitations: Models need fine-tuning, not a trading system per se
  • Integration: Could use FinGPT models for sentiment analysis within our agent

3.4 FinRobot (AI4Finance-Foundation)

  • URL: github.com/AI4Finance-Foundation/FinRobot
  • What: AI Agent platform for financial analysis using LLMs
  • Key Features: Multi-model approach (LLMs + RL + quant analytics), investment research automation
  • Maturity: Newer, building on FinGPT/FinRL
  • Integration: Worth studying for agent architecture patterns

3.5 FinMem

  • URL: github.com/pipiku915/FinMem-LLM-StockTrading
  • What: LLM trading agent with layered memory and character design
  • Key Features: Memory architecture for trading context, character-based agent design
  • Integration: Memory/context management patterns useful for our agent

3.6 StockAgent

  • URL: github.com/MingyuJ666/Stockagent
  • What: Multi-agent system simulating investor trading behavior
  • Key Features: Simulates investor decision-making with LLMs in real market conditions
  • Integration: Behavioral simulation patterns

4. Broker APIs

4.1 Alpaca

  • Type: Commission-free, API-first broker
  • API: REST + WebSocket, excellent Python SDK (alpaca-py)
  • Key Features:
    • Free stock & crypto trading (commission-free)
    • Paper trading environment
    • Real-time market data (free tier available)
    • Fractional shares
    • Simple OAuth, API key auth
  • Limitations:
    • US stocks only (no international)
    • Options support added recently (limited)
    • Smaller broker (less established)
    • Payment for order flow
  • Best For: Starting point — easiest API, free, great for development/paper trading
  • Integration: Simple REST API, perfect for an AI agent skill

4.2 Interactive Brokers (IBKR)

  • Type: Full-service institutional broker
  • API: TWS API, Client Portal API, REST (newer)
  • Key Features:
    • Global markets (stocks, options, futures, forex, bonds)
    • Professional-grade execution
    • Competitive pricing
    • Most asset classes available
    • ib_insync Python library (excellent async wrapper)
  • Limitations:
    • Complex API setup (TWS gateway must run)
    • Steep learning curve
    • Rate limits on data
    • Monthly inactivity fees (waived with $100k+)
  • Best For: Production trading — most comprehensive, institutional-grade
  • Integration: Use ib_insync for Python integration; ClawdHub already has ibkr-trader skill

4.3 Schwab (formerly TD Ameritrade)

  • Type: Major retail broker
  • API: Schwab Developer API (migrated from TDA)
  • Key Features:
    • Commission-free stock/ETF trading
    • Options trading
    • Large established broker
  • Limitations:
    • API migration from TDA has been rocky (2023-2024)
    • Less developer-friendly than Alpaca
    • OAuth flow is complex
    • Community libraries less mature post-migration
  • Best For: Users already on Schwab platform
  • Integration: Lower priority due to migration instability

Recommendation: Start with Alpaca (dev/paper), plan for IBKR (production)

5. Data Sources

5.1 Yahoo Finance (yfinance)

  • Cost: Free (unofficial API)
  • Data: Daily/intraday OHLCV, fundamentals, options chains, financials
  • Pros: Free, comprehensive, easy (yfinance Python package)
  • Cons: Unofficial (can break), rate limits, no real-time, unreliable for production
  • Best For: Prototyping, historical analysis

5.2 Polygon.io

  • Cost: Free tier (5 API calls/min, delayed), paid from $29/mo (unlimited, real-time)
  • Data: Stocks, options, forex, crypto — tick-level to daily
  • Pros: High quality, real-time WebSocket, excellent API design, options data
  • Cons: Paid for real-time, free tier very limited
  • Best For: Production data source — best balance of quality/price

5.3 Alpha Vantage

  • Cost: Free tier (25 calls/day), paid from $49.99/mo
  • Data: Stocks, forex, crypto, economic indicators, technical indicators
  • Pros: Free tier, built-in technical indicators, fundamental data
  • Cons: Severe free-tier rate limits, data quality issues reported, slow
  • Best For: Supplementary fundamental/economic data

5.4 Tiingo

  • Cost: Free tier (500 calls/hr), paid from $10/mo
  • Data: EOD stock prices, IEX real-time, news, crypto, fundamentals
  • Pros: Good free tier, reliable, clean data, news API
  • Cons: Limited intraday on free tier, smaller coverage than Polygon
  • Best For: Good budget option, especially for news data

5.5 EODHD

  • Cost: From $19.99/mo (free tier very limited)
  • Data: Global stocks, ETFs, mutual funds, bonds, fundamentals, macro
  • Pros: Global coverage, good fundamental data, affordable
  • Cons: Less real-time capability, smaller community
  • Best For: Global/fundamental data

5.6 Alpaca Market Data

  • Cost: Free with Alpaca account (IEX), paid for SIP ($9/mo)
  • Data: Real-time quotes, bars, trades via WebSocket
  • Pros: Bundled with broker, simple integration
  • Cons: IEX-only on free tier
  • Best For: When already using Alpaca for trading

Data Strategy Recommendation:

  1. Development: yfinance (free historical) + Alpaca data (free real-time IEX)
  2. Production: Polygon.io (comprehensive) + Alpaca (execution data)
  3. Supplementary: Tiingo (news), Alpha Vantage (economic indicators)

6. AI Trading Research Papers (2024-2025)

6.1 “TradingAgents: Multi-Agents LLM Financial Trading Framework” (Dec 2024)

  • Authors: Xiao, Sun, Luo, Wang (arXiv 2412.20138)
  • Key Idea: Multi-agent system mirroring real trading firms with specialized roles
  • Agents: Fundamental analyst, sentiment analyst, technical analyst, trader, risk manager
  • Approach: Debate-driven decision making among agents
  • Relevance: ⭐⭐⭐⭐⭐ — Directly applicable architecture

6.2 “Large Language Model Agent in Financial Trading: A Survey” (Jul 2024)

  • Authors: Li et al. (arXiv 2408.06361)
  • Key Idea: Comprehensive survey of LLM-based trading agents
  • Categories: News-driven, reasoning-driven, and reflection-based agents
  • Relevance: ⭐⭐⭐⭐⭐ — Essential reading for understanding the landscape

6.3 “Can Large Language Models Trade?” (Apr 2025)

  • Key Idea: Simulated stock market where LLMs act as heterogeneous competing traders
  • Finding: Tests whether LLM agents reproduce known financial theories
  • Relevance: ⭐⭐⭐⭐ — Validates LLM trading agent approach

6.4 “TradeTrap: Are LLM-based Trading Agents Truly Reliable?” (Dec 2025)

  • Key Idea: Tests reliability and faithfulness of LLM trading agents
  • Relevance: ⭐⭐⭐⭐ — Important for understanding failure modes and building safeguards

6.5 “TradingGroup: Multi-Agent Trading with Self-Reflection and Data-Synthesis” (Aug 2025)

  • Key Idea: Self-reflection mechanisms and synthetic data for trading agents
  • Relevance: ⭐⭐⭐⭐ — Self-reflection pattern directly applicable

6.6 “LLM Agents Do Not Replicate Human Market Traders” (Feb 2025)

  • Key Idea: LLM agents behave differently from human traders
  • Relevance: ⭐⭐⭐ — Important caveat for agent design

7. Synthesis & Recommendations

Architecture Decision

Build a multi-agent AI trading system combining: 1. LLM reasoning (Claude via OpenClaw) for analysis, strategy generation, risk assessment 2. Quantitative backtesting (VectorBT) for fast strategy validation 3. Structured data pipeline (yfinance + Polygon.io) for market data 4. Broker integration (Alpaca for dev, IBKR for production)

Recommended Stack

Component Tool Reason
Backtesting VectorBT Fast vectorized backtesting, parameter optimization
Historical Data yfinance + Polygon.io Free dev + quality production
Real-time Data Alpaca WebSocket Bundled with broker
Broker (Dev) Alpaca Free, simple API, paper trading
Broker (Prod) IBKR via ib_insync Comprehensive, institutional-grade
Sentiment/News Tiingo News API + LLM News feed + Claude analysis
Agent Framework Custom multi-agent (inspired by TradingAgents paper) Specialized analyst agents
Technical Analysis ta-lib or pandas-ta Standard indicators

Key Risks to Address

  1. LLM hallucination in financial reasoning — need verification layers
  2. Overfitting in backtesting — walk-forward validation essential
  3. Execution slippage — paper trading ≠ real trading
  4. Regulatory — automated trading has compliance requirements
  5. API reliability — need fallback data sources
